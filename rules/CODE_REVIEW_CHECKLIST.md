# Codex 代码审查清单

## 概述

本文档定义代码审查的标准流程和检查点，确保代码质量。

---

## 审查时机

- **每个 Phase 完成后必须进行代码审查**
- 代码提交前必须通过审查
- 重大功能开发完成后进行审查

---

## 代码逻辑审查（≥15 项检查点）

### 正确性（≥5 项）

- [ ] **逻辑错误检查**：代码逻辑是否正确，是否存在明显错误
- [ ] **边界条件检查**：边界条件是否处理正确（空值、零值、极值）
- [ ] **异常处理检查**：异常情况是否正确处理
- [ ] **状态机检查**：状态转换是否正确，状态机是否完整
- [ ] **数据类型检查**：数据类型使用是否正确，是否存在溢出风险

### 性能（≥4 项）

- [ ] **算法复杂度检查**：算法复杂度是否合理，是否存在性能瓶颈
- [ ] **资源泄漏检查**：是否存在内存泄漏、文件句柄泄漏等资源问题
- [ ] **循环优化检查**：循环是否可以优化，是否存在不必要的重复计算
- [ ] **并发安全检查**：多线程/并发场景是否安全，是否存在竞态条件

### 安全性（≥3 项）

- [ ] **注入攻击防护**：是否存在 SQL 注入、命令注入等安全漏洞
- [ ] **权限控制检查**：权限控制是否正确实现
- [ ] **敏感数据保护**：敏感数据是否妥善保护，是否存在泄露风险

### 规范（≥3 项）

- [ ] **命名规范检查**：变量、函数、文件命名是否规范
- [ ] **注释检查**：关键逻辑是否有适当的注释说明
- [ ] **代码格式检查**：代码格式是否符合项目规范

---

## 需求完成度审查（≥5 项检查点）

### 功能完整性

- [ ] **功能覆盖检查**：是否实现了规划中定义的所有功能
- [ ] **功能正确性检查**：实现的功能是否满足需求

### 测试覆盖

- [ ] **单元测试检查**：关键模块是否有单元测试
- [ ] **边界测试检查**：边界条件是否被测试覆盖

### 文档同步

- [ ] **文档更新检查**：相关文档是否已同步更新
- [ ] **接口文档检查**：接口文档是否与实际实现一致

---

## 审查流程

### 1. 提交审查

开发者完成开发后，在对话中明确要求 Codex 进行代码审查：

```
请对本次修改的代码进行 Codex 审查。
```

### 2. 执行审查

Codex 审查将按照以下清单进行检查：

```markdown
## 代码逻辑审查
### 正确性
- [ ] 检查逻辑错误
- [ ] 检查边界条件
- [ ] 检查异常处理
- [ ] 检查状态机
- [ ] 检查数据类型

### 性能
- [ ] 检查算法复杂度
- [ ] 检查资源泄漏
- [ ] 检查循环优化
- [ ] 检查并发安全

### 安全性
- [ ] 检查注入攻击防护
- [ ] 检查权限控制
- [ ] 检查敏感数据保护

### 规范
- [ ] 检查命名规范
- [ ] 检查注释
- [ ] 检查代码格式

## 需求完成度审查
- [ ] 功能覆盖检查
- [ ] 功能正确性检查
- [ ] 单元测试检查
- [ ] 边界测试检查
- [ ] 文档更新检查
- [ ] 接口文档检查
```

### 3. 反馈处理

- **CRITICAL 问题**：必须修复后才能继续
- **HIGH 问题**：建议修复
- **MEDIUM 问题**：视情况修复
- **LOW 问题**：可选修复

### 4. 复审

修复问题后，再次请求 Codex 进行复审：

```
请对修复后的代码进行复审。
```

---

## 审查通过标准

- [ ] 无 CRITICAL 问题
- [ ] HIGH 问题已修复或已记录
- [ ] 需求功能完整实现
- [ ] 文档已同步更新
- [ ] 代码符合规范

---

## 一句话总结

> **代码审查要全面，问题等级要分明，修复之后要复审。**
