# 上下文控制执行清单

## 快速决策流程

```
用户请求进来
    ↓
判断任务复杂度
    ↓
┌─────────────────┬─────────────────┬─────────────────┐
│   简单任务      │   中等任务      │   复杂任务      │
│   (<10轮)       │   (10-20轮)     │   (>20轮或      │
│                 │                 │   需大量分析)   │
└────────┬────────┴────────┬────────┴────────┬────────┘
         ↓                 ↓                 ↓
    直接处理          工具链执行        创建子Agent
         ↓                 ↓                 ↓
    返回结果          顺序调用          隔离执行
                                        ↓
                                   接收结果
                                        ↓
                                   整合输出
```

---

## 子Agent调用策略

### 1. 分析型任务（复杂代码分析）

```
任务：分析 [模块] 实现
策略：创建子Agent并行分析
输出：关键发现、风险点、优化建议
```

### 2. 研究型任务（跨模块研究）

```
任务：理解 [功能] 完整流程
策略：创建子Agent研究数据流
输出：关键发现（最多5条）、涉及文件、风险点
```

### 3. 代码生成任务（复杂功能实现）

```
任务：实现新的 [功能]
策略：创建子Agent生成代码
约束：[开发框架] 代码、文件头、错误处理、调试日志
输出：完整代码 + 使用说明
```

## 上下文清理触发器

当以下任一条件满足时执行清理：

| 触发条件 | 操作 |
|---------|------|
| 对话轮次 > 15 | 总结当前状态，询问是否继续 |
| Token 使用量 > 80% | 建议总结上下文并开启新会话 |
| 话题切换 | 明确标记"[切换话题]"，总结上一主题 |
| 任务阶段完成 | 输出"[阶段完成] 总结：..." |
| 检测到重复询问 | 立即总结已确认信息 |

## 上下文加载优先级

### P0 - 必须加载（每次会话）
- `./[文档文件名]` - 全局地图

### P1 - 任务相关（按需加载）
- 目标文件（如 `[模块目录]/[组件名]/[源文件]`）- 文件头含关键信息
- 组件 [文档文件名]（如 `[模块目录]/[组件名]/[文档文件名]`）- 第2层上下文
- 相关头文件（如 `[模块目录]/[组件名]/include/[头文件].h`）- 配置

### P2 - 深入分析（需要时加载）
- 依赖组件的实现文件
- 数据流上下游文件
- 分区表、配置文件等

## 项目特定规范

### 硬件抽象意识

1. **芯片架构**：
   - 目标芯片：[目标芯片]
   - 主频：[主频]
   - 内存：[内存大小]

2. **关键硬件模块**：
   - [硬件模块1]：[通信接口/参数]
   - [硬件模块2]：[通信接口/参数]

3. **资源限制意识**：
   - 任务栈大小：默认 2048-4096 字节
   - 堆内存：谨慎使用动态分配
   - Flash：分区表严格限制

### 核心模块意识

`[模块目录]/[核心组件]/` 是系统核心：
- 管理 [系统生命周期]
- 修改核心模块必须同步更新组件 [文档文件名] 和根目录 [文档文件名]

### 组件依赖关系

通过以下方式查看依赖：
- `[模块目录]/[组件名]/CMakeLists.txt` 的 `REQUIRES` / `PRIV_REQUIRES`
- 组件 [文档文件名] 的依赖关系章节
- 源代码中的 `#include`

**重要约束**：
- 总线共享需要互斥保护
- 避免循环依赖
- 上层可依赖下层，下层不可依赖上层

### 调试与日志

- 波特率：[波特率]
- 使用 `LOGx(TAG, fmt, ...)` 或自定义 `LOGI()` / `LOGE()`
- 调试版本：启用详细日志
- 发布版本：禁用不必要输出以节省资源

### 电源管理意识

- 超时后进入 [睡眠模式]
- 唤醒源：[唤醒源列表]
- 睡眠前需释放资源、关闭外设
- 唤醒后需重新初始化外设

## 常见任务类型处理

| 任务类型 | 建议处理方式 | 关键检查点 |
|----------|-------------|-----------|
| 修改核心模块 | 直接处理 + 更新两层 [文档文件名] | 逻辑、History、构建验证 |
| 新增组件 | 子Agent（复杂）或 直接处理 | 三层文档、CMakeLists.txt、构建验证 |
| 新增文件 | 直接处理 | 文件头、组件 [文档文件名]、构建验证 |
| 修复Bug | 直接处理 | History 记录、回归测试 |
| 优化算法 | 子Agent（如复杂） | 性能影响、栈使用、接口变化 |
| 修改配置 | 直接处理（高风险） | 配置余量、全层文档更新 |
| 代码审查 | 子Agent | 分形文档同构性 |
| 添加配置项 | 直接处理 | 配置同步到两层 [文档文件名] |

## 专项变更检查清单

### 代码变更后必须检查

```markdown
## 构建验证
- [ ] [项目构建命令] 构建成功
- [ ] 无编译警告（或已评估接受）
- [ ] 链接成功，无未定义符号

## 资源检查（如相关）
- [ ] 任务栈大小合理（建议 2048-4096）
- [ ] 任务优先级合理（避免优先级反转）
- [ ] 动态内存使用可控
- [ ] Flash 使用量在分区余量内

## 功能检查（如相关）
- [ ] 睡眠/唤醒路径正确
- [ ] 中断处理安全
- [ ] 外设初始化和释放配对
- [ ] 日志关键路径已验证

## 配置检查（如相关）
- [ ] 配置文件同步到 [文档文件名]
- [ ] 默认值合理
```

### 任务修改专项检查

```markdown
## 新增/修改任务
- [ ] 任务栈大小：起始 2048，根据调用链深度调整
- [ ] 任务优先级：合理范围，避免与系统任务冲突
- [ ] 组件 [文档文件名] 记录任务配置（栈大小、优先级、职责）
- [ ] 任务退出/删除逻辑正确
- [ ] 任务间同步机制（信号量/队列）正确
```

## 分形文档同构快速检查

### 修改文件后立即执行

```markdown
## 第3层（文件头）
- [ ] History 添加了本次变更条目
- [ ] Description 与实际代码一致

## 第2层（组件 [文档文件名]）
- [ ] 成员清单包含本文件（如新增）
- [ ] 接口契约与实际导出一致（如接口变化）
- [ ] REQUIRES/PRIV_REQUIRES 准确（如依赖变化）

## 第1层（根目录 [文档文件名]）
- [ ] 模块状态准确（如重大变更）
- [ ] 最近变更已记录（如架构影响）
```

## 一句话原则

**小任务自己干，大任务拆给子Agent，保持主上下文干净。**

**专项：文档同构优先，变更后必构建，资源限制常记心。**
