# 通用项目初始化规范创建计划

## 概述

深入理解当前项目的开发规范，形成一套可复用的、通用的、用于初始化其他项目的规范。

## 背景

当前项目（planck Mini & Air 智能牙刷固件）已经建立了一套成熟的开发规范，核心包括：
- 分形文档（三层文档结构）
- 回环同构（代码变更后必更新文档）
- Codex 代码审查
- 阶段规划文档更新
- 阶段代码提交

## 目标

创建通用规范，使得这些实践可以快速部署到新项目中。

### 核心产出物（精简版）

初始化一个新项目时，只需生成两个产物：

1. **CLAUDE.md** - 项目根目录入口文档
   - 包含项目概述、构建命令、模块结构、开发规范入口

2. **/rules/** 文件夹 - 开发规范集合（5 个文档）
   - FRACTAL_DOC.md - 分形文档规范
   - CONTEXT_CHECKLIST.md - 回环检查清单
   - FRACTAL_DOC_PROTOCOL.md - 执行协议
   - PHASE_MANAGEMENT.md - 阶段管理规范
   - CODE_REVIEW_CHECKLIST.md - Codex 审查清单

**不需要**：复杂的初始化脚本、多种环境的模板、大量工具。只需规范文档即可。

---

## 术语表

| 术语 | 定义 | 备注 |
|-----|------|------|
| 分形文档 | 三层结构的项目文档体系 | 第1层→第2层→第3层 |
| 回环同构 | 代码变更后文档同步更新 | 确保文档与代码一致 |
| Codex 审查 | 代码审查流程 | 使用 code-reviewer agent |
| Phase | 项目实施阶段 | 建议 3-7 个 |
| 准入标准 | 进入下一阶段前必须满足的条件 | 构建成功、审核通过 |
| 准出标准 | 完成当前阶段必须满足的条件 | 文档更新、代码提交 |
| 文件头 | 源文件开头的元信息区域 | 包含 History、Description 等 |

---

## 可配置项

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| 模块目录名 | components/ | 可改为 src/, lib/ 等 |
| 文档文件名 | CLAUDE.md | 可改为 README.md 等，模板中用占位符 [文档文件名] 表示 |
| 规则目录名 | rules/ | 存放规范定义 |
| 规划目录名 | plans/ | 存放项目规划 |
| 文件头格式 | 公司模板 | 可简化或替换 |
| 审查工具 | code-reviewer | 可替换为其他审查工具 |
| 文档更新时机 | 每次代码变更 | 强制规则：任何代码变更必须触发文档更新<br>允许优化的场景（需同时满足以下条件）：<br>1. 仅注释/格式调整的微小变更<br>2. 单日累计变更不超过 5 个文件<br>3. 变更记录到 CHANGELOG.md<br>4. 次日 12:00 前完成汇总更新 |

## 实施步骤

### Phase 1: 分析现有规则并抽象化

**目标：** 理解现有规则，提取可复用部分

**项目管理要素：**
- **预计工时**：2-4 小时
- **输入**：现有 /rules/ 目录下的所有 .md 文件、现有 CLAUDE.md
- **输出**：抽象化对照表（项目特定 vs 通用模式）
- **准入标准**：理解现有规则内容
- **准出标准**：完成抽象化分析

**任务：**
- [ ] 1.1 分析现有规则
      - FRACTAL_DOC.md：提取三层分形结构
      - CONTEXT_CHECKLIST.md：提取回环检查流程
      - FRACTAL_DOC_PROTOCOL.md：提取执行协议
- [ ] 1.2 抽象化为通用模式
      - 将 build 命令抽象为"构建验证"
      - 将 idf.py 抽象为"项目构建工具"
      - 将 components/ 抽象为"[模块目录]"

**交付物：**
- 抽象化对照表（项目特定 vs 通用模式）

---

### Phase 2: 创建通用 CLAUDE.md 模板

**目标：** 创建可复用的项目根文档模板

**项目管理要素：**
- **预计工时**：2-4 小时
- **输入**：抽象化对照表、现有 CLAUDE.md
- **输出**：CLAUDE.md 模板
- **准入标准**：抽象化分析完成
- **准出标准**：模板包含所有必要章节

**任务：**
- [ ] 2.1 创建 CLAUDE.md 模板结构
      - 项目概述
      - 构建命令（抽象化占位符）
      - 模块结构表
      - 架构说明
      - 开发规范入口（指向 /rules/）
      - 最近变更记录

**交付物：**
- CLAUDE.md 模板（通用版本）

---

### Phase 3: 创建 /rules/ 规范文件夹

**目标：** 创建可复用的规则集合

**项目管理要素：**
- **预计工时**：4-8 小时
- **输入**：抽象化对照表、现有规则
- **输出**：完整的 /rules/ 文件夹
- **准入标准**：CLAUDE.md 模板完成
- **准出标准**：规则文档完整可用

**任务：**
- [ ] 3.1 创建 FRACTAL_DOC.md - 分形文档规范（通用版）
- [ ] 3.2 创建 CONTEXT_CHECKLIST.md - 回环检查清单（通用版）
- [ ] 3.3 创建 FRACTAL_DOC_PROTOCOL.md - 执行协议（通用版）
- [ ] 3.4 创建 PHASE_MANAGEMENT.md - 阶段管理规范
- [ ] 3.5 创建 CODE_REVIEW_CHECKLIST.md - Codex 审查清单

**交付物：**
- /rules/ 文件夹（含 5 个规则文档）

## 核心要素说明

### 初始化产出物（精简版）

```
新项目初始化后
├── CLAUDE.md              ← 第1层：全局地图
├── [模块目录]/
│   ├── CLAUDE.md          ← 第2层：模块清单
│   └── [源文件]           ← 第3层：文件头
└── rules/
    ├── FRACTAL_DOC.md     ← 分形文档规范
    ├── CONTEXT_CHECKLIST.md ← 回环检查清单
    ├── FRACTAL_DOC_PROTOCOL.md ← 执行协议
    ├── PHASE_MANAGEMENT.md ← 阶段管理规范
    └── CODE_REVIEW_CHECKLIST.md ← Codex 审查清单
```

### 1. 分形文档（三层结构）

```
项目根目录
├── [文档文件名]         ← 第1层：全局地图
├── [模块目录]/
│   ├── [文档文件名]     ← 第2层：模块清单
│   └── [源文件]        ← 第3层：文件头
└── rules/
    └── [规范文件]       ← 规范定义
```

### 2. 回环同构

```
代码变更
    ↓
更新第3层（文件头 History）
    ↓
更新第2层（如成员/接口/依赖变化）
    ↓
更新第1层（如架构/模块变化）
    ↓
构建验证（如项目构建工具）
    ↓
自检同构
```

### 3. Codex 审查

每个 Phase 完成后：
- 代码逻辑审查（正确性、性能、安全性）
- 需求完成度审查
- 审核通过后方可提交

### 4. 规划文档

- 存放在 `/plans/` 目录
- 每个 Phase 更新进度
- 记录决策和变更

### 5. 阶段提交

- 每个 Phase 完成后提交代码
- 遵循 git workflow 规范
- 包含详细的变更说明

---

## 依赖关系

- Phase 1 → Phase 2（抽象化分析完成后才能创建模板）
- Phase 2 → Phase 3（模板完成后才能创建规则文件夹）

## 风险评估

| 风险 | 等级 | 触发信号 | 责任人 | 缓解措施 | 应急措施 | 复盘时间 |
|------|------|----------|--------|----------|----------|----------|
| 过度抽象导致不实用 | 中 | 模板使用者反馈无法适配实际项目 | 项目负责人 | 保留示例，保持可操作性；提供可配置项 | 简化抽象，恢复项目特定内容 | 每周站会 |
| 与现有规则冲突 | 低 | 规则检查报错或 lint 失败 | 规则维护者 | 明确适用范围，避免覆盖项目特定规则 | 调整规则优先级或创建项目特定覆盖 | 每次规则变更后 |
| 抽象过程中丢失关键实践细节 | 中 | 模板缺少关键步骤或说明不清晰 | 项目负责人 | 每个核心要素保留详细说明和示例 | 补充遗漏内容，恢复详细说明 | 每次文档产出后 |
| 验证周期过长导致计划搁置 | 高 | Phase 进度连续两周无更新 | 项目负责人 | 设定明确的时间盒，每个 Phase 不超过 1 周 | 缩减范围，优先完成核心功能 | 每周 |

## 验收标准

- [ ] CLAUDE.md 模板包含所有必要章节：项目概述、构建命令、模块结构、架构说明、开发规范入口、最近变更记录（对应验证1-V01）
- [ ] /rules/ 文件夹包含 5 个规则文档：FRACTAL_DOC.md、CONTEXT_CHECKLIST.md、FRACTAL_DOC_PROTOCOL.md、PHASE_MANAGEMENT.md、CODE_REVIEW_CHECKLIST.md（对应验证2-V02）
- [ ] 回环检查清单包含 ≥15 项检查点（对应验证2-V03）
- [ ] Codex 审查清单包含 ≥15 项检查点（对应验证2-V04）
- [ ] 阶段管理规范包含：阶段划分原则、规划文档更新要求、代码提交时机（对应验证2-V05）

## 验证方法

### 验证1：模板可用性测试
- **对应验收标准**：
  - V01: CLAUDE.md 模板包含所有必要章节
- **测试方法**：
  1. 检查 CLAUDE.md 模板结构
  2. 确认包含：项目概述、构建命令、模块结构、架构说明、开发规范入口、最近变更记录
- **评估标准**：
  - V01: 模板章节完整（6 个章节），可直接用于新项目

### 验证2：规则完整性测试
- **对应验收标准**：
  - V02: /rules/ 文件夹包含 5 个规则文档
  - V03: 回环检查清单 ≥15 项
  - V04: Codex 审查清单 ≥15 项
  - V05: 阶段管理规范完整
- **测试方法**：
  1. 检查 /rules/ 文件夹中是否存在 5 个规则文档：FRACTAL_DOC.md、CONTEXT_CHECKLIST.md、FRACTAL_DOC_PROTOCOL.md、PHASE_MANAGEMENT.md、CODE_REVIEW_CHECKLIST.md
  2. 统计回环检查清单（CONTEXT_CHECKLIST.md）的检查点数量
  3. 统计 Codex 审查清单（CODE_REVIEW_CHECKLIST.md）的检查点数量
  4. 检查阶段管理规范（PHASE_MANAGEMENT.md）是否包含阶段划分原则、规划文档更新要求、代码提交时机
- **评估标准**：
  - V02: /rules/ 文件夹包含且仅包含 5 个指定规则文档
  - V03: 回环检查清单检查点数量 ≥ 15
  - V04: Codex 审查清单检查点数量 ≥ 15
  - V05: 阶段管理规范包含 3 个必要章节
