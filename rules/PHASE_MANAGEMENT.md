# 阶段管理规范

## 概述

本文档定义项目开发阶段（Phase）的划分原则、管理流程和交付标准。

---

## 阶段划分原则

### 1. 边界清晰，可独立交付

每个 Phase 应该有：
- 明确的起始和结束条件
- 独立的交付物
- 可独立测试验证

### 2. 建议 3-7 个 Phase

- **小型项目（< 1个月）**：3-4 个 Phase
- **中型项目（1-3个月）**：4-5 个 Phase
- **大型项目（> 3个月）**：5-7 个 Phase

### 3. Phase 划分参考

| Phase | 内容 | 典型交付物 |
|-------|------|-----------|
| Phase 1 | 需求分析/架构设计 | 规划文档、架构图 |
| Phase 2 | 核心功能开发 | 核心模块代码 |
| Phase 3 | 功能完善 | 功能模块、单元测试 |
| Phase 4 | 集成测试 | 集成测试报告 |
| Phase 5 | 优化/发布 | 性能优化、发布版本 |

---

## 规划文档更新要求

### 1. 创建规划文档

每个项目应在 `/plans/` 目录下创建规划文档，包含：

```markdown
# [项目名称] 实施计划

## 概述
[项目目标]

## 背景
[项目背景]

## 目标
[具体目标]

## 实施步骤

### Phase 1: [名称]
**目标：** [阶段目标]

**任务：**
- [ ] 任务1
- [ ] 任务2

**交付物：**
- [交付物1]
- [交付物2]

### Phase 2: [名称]
...

## 验收标准
 标准- [ ]1
- [ ] 标准2
```

### 2. 更新进度

每个 Phase 完成后必须更新：
- 当前进度（已完成/进行中）
- 实际交付物
- 遇到的问题和解决方案
- 下一步计划

### 3. 记录决策和变更

规划文档应记录：
- 关键设计决策及原因
- 需求变更及影响
- 技术选型变更
- 进度偏差及调整

---

## 代码提交时机

### 1. 提交规则

- **每个 Phase 完成后提交代码**
- 遵循 git workflow 规范
- 包含详细的变更说明

### 2. 提交信息格式

```
[类型]: [简短描述]

[详细描述]

相关Phase: Phase X
```

类型：feat, fix, refactor, docs, test, chore, perf, ci

### 3. 提交前检查

- [ ] 代码通过 Codex 审核
- [ ] 分形文档已更新（同构检查通过）
- [ ] 构建验证通过
- [ ] 无未解决的 lint 问题

## Phase 管理流程

### Phase 启动

1. **阅读规划文档**
   - 确认 Phase 目标和任务
   - 了解验收标准

2. **创建任务列表**
   - 分解为可执行的任务
   - 设定优先级

3. **准备环境**
   - 确保开发环境就绪
   - 获取必要的资源

### Phase 执行

1. **开发任务**
   - 按照任务列表执行
   - 保持文档同步更新

2. **自检**
   - 对照验收标准自检
   - 执行分形文档同构检查

3. **代码审核**
   - 调用 Codex 进行审核
   - 修复审核发现的问题

### Phase 交付

1. **更新规划文档**
   - 记录完成状态
   - 添加交付物清单

2. **提交代码**
   - 确保所有检查通过
   - 编写详细的提交信息

3. **进入下一 Phase**
   - 确认上一 Phase 已完成
   - 启动下一 Phase

---

## 准入/准出标准

### 准入标准（进入 Phase 前）

- [ ] 上一 Phase 已完成并通过验收
- [ ] 规划文档已更新
- [ ] 开发环境就绪

### 准出标准（完成 Phase 前）

- [ ] 所有任务已完成
- [ ] 交付物已产出
- [ ] Codex 审核通过
- [ ] 分形文档已更新
- [ ] 代码已提交

---

## 风险管理

### 常见风险

| 风险 | 应对措施 |
|------|----------|
| 进度延期 | 及时调整 Phase 边界，优先保证核心功能 |
| 需求变更 | 记录变更，评估影响，更新规划文档 |
| 技术难题 | 预留缓冲时间，必要时寻求帮助 |
| 资源不足 | 提前规划，优先级排序 |

### 风险识别时机

- 每日站会
- Phase 边界检查
- 定期回顾

---

## 一句话总结

> **Phase 有边界，交付有标准，变更有记录，提交有审核。**
