# 分形文档架构 + 强制同构 执行协议

## 核心思想

> **文档和代码是同一事物的两个相，必须保持同构。**

通过三层分形文档结构，确保 AI 从任意文件进入，三次 Read 就能拿到完整上下文。

---

## 三层文档结构（分形）

```
📁 项目根目录
├── 📄 [文档文件名]           ← 第1层：全局地图
├── 📁 [模块目录]
│   ├── 📁 [组件名]
│   │   ├── 📄 [文档文件名]  ← 第2层：模块清单（必须存在）
│   │   ├── 📄 [源文件]
│   │   ├── 📄 include/[头文件].h
│   │   └── 📄 CMakeLists.txt
│   └── ...
├── 📁 main
│   ├── 📄 [文档文件名]       ← 第2层（入口组件）
│   └── 📄 main.c             ← 第3层
├── 📄 CMakeLists.txt         ← 项目构建配置
├── 📄 Kconfig.projbuild      ← 项目配置
├── 📄 [分区表文件]           ← 分区表
└── 📁 rules
    └── 📄 FRACTAL_DOC_PROTOCOL.md ← 本文档
```

### 第1层：根目录 [文档文件名] - 全局地图

**位置：** `./[文档文件名]`（相对路径，非绝对路径）

**职责：** 项目整体导航

**内容：**
```markdown
# [文档文件名]

## 项目概述
- 名称：[项目名称]
- 技术栈：[技术栈描述]
- 目标：[项目目标]

## 模块结构
| 模块 | 路径 | 核心职责 | 状态 |
|------|------|----------|------|
| [组件名] | [模块目录]/[组件名] | [职责] | ✅ 稳定 |
| ... | ... | ... | ... |

## 关键配置文件
- [分区表文件] - 分区表配置
- sdkconfig - 编译配置
- Kconfig.projbuild - 项目自定义配置
- version.txt - 版本定义

## 快速导航
- 接口规范：见各组件 [文档文件名]
- 核心模块：见 [模块目录]/[核心组件]/[文档文件名]

## 最近变更
- [YYYY-MM-DD]: [变更说明]
```

### 第2层：组件目录 [文档文件名] - 模块清单

**位置：** `[模块目录]/[组件名]/[文档文件名]`

**职责：** 单个组件的内部结构、接口、依赖

**内容：**
```markdown
# 模块：[组件名]

## 模块职责
[一句话描述模块的核心职责]

## 成员清单
| 文件 | 职责 | 对外接口 |
|------|------|----------|
| [源文件] | 核心实现 | init(), process() |
| include/[头文件].h | 接口定义 | 宏定义、函数声明 |
| CMakeLists.txt | 构建配置 | - |

## 对外接口
### 函数名(参数)
- 输入：[参数说明]
- 输出：[返回值说明]
- 异常：[可能的错误]

## 依赖关系
- **REQUIRES:** [本组件依赖的其他组件]
- **PRIV_REQUIRES:** [本组件的私有依赖]
- **被依赖：** [依赖本组件的其他组件]

## Kconfig 配置
- CONFIG_[组件名]_[选项名] - [选项说明]

## 变更记录
- [YYYY-MM-DD]: [变更说明]
```

### 第3层：文件头 - 代码级声明

**位置：** 每个源文件和头文件的头部

**职责：** 单个文件的元信息

**Date 字段说明：**
- `Date` - 文件**创建日期**（首次创建时填写，保持不变）
- `History` - 记录所有**变更历史**（每次修改必须添加条目）

**格式：**
```c
/********************************Copyright (c)***********************************
 **
 **                     Copyright (c), [年份] , [公司/作者]
 **                            All Rights Reserved
 **
 **---------------------------------File Info------------------------------------
 ** FileName:    [文件名]
 ** Author:      [作者]
 ** Version:     [版本号]
 ** Date:        [创建日期]        ← 创建日期，保持不变
 ** Description: [职责描述]
 **
 ** History:
 **      1.Date:         [变更日期]
 **        Author:       [变更作者]
 **        Modification: [变更说明]
 **      2.Date:         [创建日期]
 **        Author:       [作者]
 **        Modification: 创建文件
 *******************************************************************************/
```

## 强制同构规则

### 规则1：变更必回环

```
代码变更
    ↓
更新第3层（文件头 History - 任何功能、行为、时序、资源占用变化）
    ↓
更新第2层（组件 [文档文件名]，如成员/接口/REQUIRES/Kconfig变化）
    ↓
更新第1层（根目录 [文档文件名]，如架构/模块状态/配置变化）
    ↓
构建验证（[项目构建命令]）
```

### 规则2：三层一致性检查

| 层级 | 检查点 | 检查内容 | 时机 |
|------|--------|----------|------|
| 第3层 | 文件头 | Description 是否正确 | 每次修改文件 |
| 第3层 | 文件头 | History 是否记录最新变更 | 每次修改文件 |
| 第3层 | 文件头 | Date 是否为创建日期 | 文件创建时 |
| 第2层 | [文档文件名] | 成员清单是否完整 | 新增/删除文件 |
| 第2层 | [文档文件名] | 接口契约是否与实际一致 | 修改对外接口 |
| 第2层 | [文档文件名] | 依赖关系（REQUIRES）是否准确 | 修改依赖时 |
| 第2层 | [文档文件名] | Kconfig 配置是否同步 | 修改 Kconfig 时 |
| 第1层 | [文档文件名] | 模块结构表是否完整 | 新增/删除组件 |
| 第1层 | [文档文件名] | 模块状态是否准确 | 模块重大变更 |
| 第1层 | [文档文件名] | 配置变更是否记录 | 修改配置文件 |

### 规则3：异常即阻断（非紧急修复场景）

- 发现文档与代码不一致 → **阻断合并到主分支**
- 紧急修复分支可先开发 → 合并前必须修复文档同构性
- 先修复文档 → 再修改代码（或同步修改）

## Claude Code 执行逻辑

### 当读取任意文件时：

```
读取 [模块目录]/[组件名]/[源文件]
  ↓
看到文件头（第3层）→ 知道职责、依赖、变更历史
  ↓
读取 [模块目录]/[组件名]/[文档文件名]（第2层）→ 了解模块全貌、接口契约、REQUIRES
  ↓
读取 ./[文档文件名]（第1层）→ 掌握全局架构
  ↓
三次 Read，完整上下文
```

### 当修改代码时：

```
1. 修改代码前
   → 读取文件头确认当前状态（第3层）
   → 读取组件 [文档文件名] 了解模块上下文（第2层）

2. 修改代码后
   → 更新文件头 History（第3层）- 任何功能变更都记录

3. 检查第2层
   → 成员清单是否需要更新？
   → 接口契约是否变化？
   → REQUIRES/PRIV_REQUIRES 是否变化？
   → Kconfig 配置是否同步？

4. 检查第1层
   → 架构地图是否需要更新？
   → 模块状态是否需要更新？
   → 配置是否需要更新？

5. 构建验证
   → [项目构建命令] 成功？
   → 任务栈/优先级合理？

6. 执行同构检查（对照自检清单）
```

## 自检清单

```markdown
# 同构检查清单

## 第3层检查（每次修改文件）
- [ ] 文件头 Description 与实际代码职责一致
- [ ] 文件头 History 包含本次变更记录（任何功能/行为/时序/资源变更）
- [ ] 文件头 Date 为创建日期（非修改日期）
- [ ] 新增/删除的 #include 依赖正确

## 第2层检查（文件增删时）
- [ ] 组件 [文档文件名] 成员清单包含所有文件
- [ ] 组件 [文档文件名] 接口契约与实际 export 一致
- [ ] 组件 [文档文件名] 依赖关系（REQUIRES/PRIV_REQUIRES）准确
- [ ] CMakeLists.txt 正确配置了源文件列表和依赖
- [ ] 新增文件已添加文件头

## 第1层检查（架构变更时）
- [ ] 根目录 [文档文件名] 模块结构表完整
- [ ] 根目录 [文档文件名] 模块状态标注准确
- [ ] 跨模块依赖在全局地图中体现
- [ ] 配置变更在全局地图中记录
- [ ] 根目录 [文档文件名] 最近变更已记录

## 构建检查（修改后必须执行）
- [ ] [项目构建命令] 构建成功
- [ ] 如有配置修改，检查配置同步
- [ ] 如修改分区表，检查固件余量
- [ ] 如新增任务，检查栈大小和优先级
```

## 快速开始模板

### 创建新组件（三层完整创建）

```bash
# 1. 创建组件目录
mkdir -p [模块目录]/[新组件]/include

# 2. 创建第2层（组件 [文档文件名]）
cat > [模块目录]/[新组件]/[文档文件名] << EOF
# 模块：[组件名]

## 模块职责
[一句话描述核心职责]

## 成员清单
| 文件 | 职责 | 对外接口 |
|------|------|----------|
| [源文件] | 核心实现 | init(), process() |

## 对外接口
### init(void)
- 功能：初始化模块
- 调用时机：系统启动时

## 依赖关系
- **REQUIRES:** [依赖的组件]
- **PRIV_REQUIRES:** [私有依赖]
- **被依赖：** [被哪些组件依赖]

## Kconfig 配置
- [如有配置项请列出]

## 变更记录
- YYYY-MM-DD: 创建模块
EOF

# 3. 创建第3层（源文件）- 手动填入实际日期
cat > [模块目录]/[新组件]/[源文件] << EOF
/********************************Copyright (c)***********************************
 ** FileName:    [源文件]
 ** Author:      [作者]
 ** Version:     V1.0
 ** Date:        YYYY-MM-DD       ← 填入实际创建日期
 ** Description: [职责描述]
 **
 ** History:
 **      1.Date:         YYYY-MM-DD
 **        Author:       [作者]
 **        Modification: 创建文件
 *******************************************************************************/

#include "[头文件].h"
// ...
EOF

# 4. 更新第1层（根目录 [文档文件名]）
# 手动编辑，在模块结构表中添加新组件

# 5. 构建验证
[项目构建命令] set-target [目标芯片] && [项目构建命令] build
```

## 关键文件映射

| 文件/目录 | 所属层级 | 影响的文档 |
|-----------|----------|-----------|
| `[模块目录]/[组件名]/[源文件]` | 第3层 | 同目录 [文档文件名]（第2层） |
| `[模块目录]/[组件名]/[文档文件名]` | 第2层 | 根目录 [文档文件名]（第1层） |
| `main/*` | 第2/3层 | 根目录 [文档文件名]（入口说明） |
| `[核心组件]/*` | 第2/3层 | 根目录 [文档文件名] 核心章节 |
| `[配置文件]` | 第1层 | 根目录 [文档文件名] 配置章节 |

## 构建配置模板

### 组件 CMakeLists.txt 模板

```cmake
idf_component_register(
    SRCS "[源文件]" "[工具文件]"
    INCLUDE_DIRS "include"
    REQUIRES [公共依赖]
    PRIV_REQUIRES [私有依赖]
)
```

### 配置同步

修改 `Kconfig.projbuild` 后，必须同步更新：
1. 组件 [文档文件名] 的配置章节
2. 根目录 [文档文件名] 的配置说明

## 一句话总结

> **三层文档，三次 Read，同构一致，变更回环。**

分形文档让 AI 快速建立上下文，强制同构确保上下文不腐烂。

**核心原则：**
- 第3层变更（任何功能/行为/时序/资源变更） → 必更新文件头 History
- 第2层变更（成员/接口/REQUIRES） → 必检查组件 [文档文件名]
- 第1层变更（架构/模块/配置） → 必检查根目录 [文档文件名]
- 变更后 → 必执行构建验证
