# 分形文档执行指令

## 核心原则

> **文档和代码是同一事物的两个相，必须保持同构。**

通过三层分形文档结构，确保 AI 从任意文件进入，三次 Read 就能拿到完整上下文。

---

## 三层分形结构

```
📁 项目根目录
├── 📄 [文档文件名]        ← 第1层：全局地图（项目概述、构建命令、模块索引）
├── 📁 [模块目录]
│   ├── 📁 [组件名]
│   │   ├── 📄 [文档文件名] ← 第2层：模块清单（成员清单、接口契约、依赖关系）
│   │   ├── 📄 [源文件]    ← 第3层：源代码（含文件头）
│   │   ├── 📄 include/[头文件].h
│   │   └── 📄 CMakeLists.txt
│   └── ... (其他组件)
├── 📁 main
│   ├── 📄 main.c          ← 第3层（入口文件）
│   └── 📄 [文档文件名]     ← 第2层（main组件文档）
├── 📄 CMakeLists.txt      ← 第1层（项目构建配置）
├── 📄 Kconfig.projbuild   ← 第1层（项目配置）
├── 📄 [分区表文件]        ← 第1层（分区表）
└── 📁 rules
    └── 📄 FRACTAL_DOC.md  ← 本文档
```

### 第1层：根目录 [文档文件名] - 全局地图

**职责：** 项目整体导航，提供全局视角

**必须包含：**
- 项目概述（名称、技术栈、目标）
- 构建命令
- 模块结构表（所有组件及入口文件列表）
- 关键配置文件说明
- 快速导航指引

### 第2层：组件目录 [文档文件名] - 模块清单

**职责：** 描述单个组件的内部结构、接口契约、依赖关系

**位置：** `[模块目录]/[组件名]/[文档文件名]`

**必须包含：**
- 模块职责（一句话描述）
- 成员清单（文件列表及职责）
- 对外接口（函数/宏定义）
- 依赖关系（REQUIRES/PRIV_REQUIRES）
- Kconfig 配置项（如有）
- 变更记录

### 第3层：文件头 - 代码级声明

**职责：** 单个文件的元信息、依赖、职责、变更历史

**格式：** 项目标准文件头

**Date 字段说明：**
- `Date` - 文件**创建日期**（首次创建时填写，后续不修改）
- `History` - 记录所有**变更历史**（每次修改必须添加条目）

## 收到用户请求时的执行流程

### 第一步：定位上下文（3次Read）

```
如果用户提到具体文件：
  Read 文件 → 看到文件头（第3层）
  Read 同目录 [文档文件名]（第2层）→ 了解模块全貌
  Read 根目录 [文档文件名]（第1层）→ 掌握全局架构

如果用户提到模块：
  Read 组件 [文档文件名]（第2层）
  Read 根目录 [文档文件名]（第1层）
  按需读取具体文件（第3层）

如果用户全局问题：
  Read 根目录 [文档文件名]（第1层）
  按需深入模块（第2层）
```

### 第二步：执行变更（强制回环）

```python
# 修改代码前
read(file_path)  # 确认当前状态，查看文件头（第3层）

# 修改代码
edit(file_path, ...)

# 修改后检查（必须执行）
updated_content = read(file_path)

# 1. 更新第3层（文件头）- 任何功能、行为、时序、资源占用变化都更新
if 代码行为变化 or 接口变化 or 依赖变化 or 时序变化 or 资源占用变化:
    update_file_header(file_path)  # 添加 History 条目

# 2. 更新第2层（组件 [文档文件名]）
module_doc = read([模块目录] + "/[组件名]/[文档文件名]")
if 成员清单需更新 or 接口契约需更新 or REQUIRES变化 or PRIV_REQUIRES变化 or Kconfig变化:
    edit(module_doc, 更新成员清单或接口契约或依赖关系或Kconfig配置)

# 3. 更新第1层（根目录 [文档文件名]）
if 架构影响 or 新增组件 or 模块状态变化 or 分区表变化 or Kconfig变化:
    global_doc = read("./[文档文件名]")
    edit(global_doc, 更新模块结构表或分区配置或Kconfig说明)

# 4. 构建验证
run_build_check()

# 5. 自检
run_self_check()
```

## 代码修改场景模板

### 场景1：修改现有函数

```python
# 1. 读取确认（3层）
read("[模块目录]/[组件名]/[源文件]")      # 第3层
read("[模块目录]/[组件名]/[文档文件名]")   # 第2层
read("./[文档文件名]")                   # 第1层

# 2. 修改代码
edit("[模块目录]/[组件名]/[源文件]", old_func, new_func)

# 3. 更新第3层（文件头）- 任何功能变更都添加 History
# 添加 History 条目：日期、作者、变更说明

# 4. 更新第2层（如接口变化）
# 如对外接口变化，更新 [文档文件名] 接口契约表

# 5. 更新第1层（如架构影响）
# 如修改核心模块，更新根目录 [文档文件名] 描述

# 6. 构建验证
run("[项目构建命令]")
```

### 场景2：新增文件

```python
# 1. 创建文件（必须带标准文件头）- 第3层
write("[模块目录]/[组件名]/[新文件]", '''
/********************************Copyright (c)***********************************
 ** FileName:    [文件名]
 ** Author:      [作者]
 ** Version:     V1.0
 ** Date:        [YYYY-MM-DD]    ← 创建日期
 ** Description: [职责描述]
 ** History:
 **      1.Date: [YYYY-MM-DD] Author: [作者] Modification: 创建文件
 *******************************************************************************/
''')

# 2. 更新第2层（组件 [文档文件名]）
read("[模块目录]/[组件名]/[文档文件名]")
edit([文档文件名], 成员清单, 添加新文件)

# 3. 更新构建配置
# 更新 CMakeLists.txt 的 SRCS

# 4. 如有新接口，更新接口契约
```

### 场景3：新增组件

```python
# 1. 创建组件目录和 [文档文件名] - 第2层
write("[模块目录]/[新组件]/[文档文件名]", '''
# 模块：[组件名]

## 模块职责
[一句话描述]

## 成员清单
| 文件 | 职责 | 对外接口 |
|------|------|----------|
| [源文件] | 核心实现 | init(), process() |

## 对外接口
### init(void)
- 功能：初始化模块

## 依赖关系
- **REQUIRES:** [依赖的组件]
- **PRIV_REQUIRES:** [私有依赖]
- **被依赖：** [被哪些组件依赖]

## Kconfig 配置
- [如有配置项请列出]

## 变更记录
- [YYYY-MM-DD]: 创建模块
''')

# 2. 创建源文件 - 第3层
write("[模块目录]/[新组件]/[源文件]", [带文件头])
write("[模块目录]/[新组件]/include/[头文件].h", [带文件头])
write("[模块目录]/[新组件]/CMakeLists.txt", [构建配置])

# 3. 更新第1层（根目录 [文档文件名]）
read("./[文档文件名]")
edit([文档文件名], 模块结构表, 添加新组件)

# 4. 构建验证
run("[项目构建命令] set-target [目标芯片] && [项目构建命令] build")
```

### 场景4：删除文件

```python
# 1. 删除文件

# 2. 更新第2层（组件 [文档文件名]）
read("[模块目录]/[组件名]/[文档文件名]")
edit([文档文件名], 成员清单, 移除文件)

# 3. 更新 CMakeLists.txt

# 4. 如删除整个组件，更新第1层
read("./[文档文件名]")
edit([文档文件名], 模块结构表, 移除组件)
```

## 自检清单（每次修改后执行）

```markdown
## 第3层检查（文件级）
- [ ] 文件头 Description 与实际代码职责一致
- [ ] 文件头 History 记录了本次变更（任何功能变更都必须记录）
- [ ] 文件头 Date 为创建日期（非修改日期）
- [ ] 新增/删除的 #include 依赖正确

## 第2层检查（模块级）
- [ ] [文档文件名] 成员清单包含所有文件
- [ ] [文档文件名] 接口契约与实际 export 一致
- [ ] [文档文件名] 依赖关系（REQUIRES/PRIV_REQUIRES）准确
- [ ] CMakeLists.txt 正确配置了源文件列表
- [ ] 新增文件已添加文件头

## 第1层检查（全局级）
- [ ] 根目录 [文档文件名] 模块结构表完整
- [ ] 新增组件已在模块结构表中登记
- [ ] 模块状态标注准确（开发中/稳定/废弃）
- [ ] 最近变更已记录

## 构建检查（修改后必须执行）
- [ ] [项目构建命令] 构建成功
- [ ] 如有 Kconfig 修改，检查配置同步
- [ ] 如修改分区表，检查固件余量
- [ ] 如新增任务，检查栈大小和优先级
```

## 用户提示词识别

| 关键词 | 动作 |
|--------|------|
| "新增模块" | 创建组件目录 + [文档文件名]（第2层）+ 更新根目录 [文档文件名]（第1层）+ 构建验证 |
| "创建文件" | 必须添加文件头（第3层）+ 更新组件 [文档文件名]（第2层） |
| "修改接口" | 更新文件头 History（第3层）+ 更新组件 [文档文件名] 接口契约（第2层） |
| "重构" | 全面检查三层文档同构性 + 构建验证 |
| "删除文件" | 删除文件 + 更新组件 [文档文件名]（第2层）+ 如需要更新根目录 [文档文件名]（第1层） |

---

## 工作流总结

```
1. Read 3次 → 建立完整上下文（第3层 → 第2层 → 第1层）
2. Modify 代码 → 实现需求
3. Update 第3层 → 更新文件头 History（任何功能变更）
4. Update 第2层 → 如有需要更新组件 [文档文件名]
5. Update 第1层 → 如有需要更新根目录 [文档文件名]
6. Build 检查 → [项目构建命令] 成功
7. Check 同构 → 确保三层一致
8. Report → 告知完成
```

**核心原则：三层文档，三次 Read，同构一致，变更回环。**
